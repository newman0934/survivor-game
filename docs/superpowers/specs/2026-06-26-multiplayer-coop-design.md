# 設計文件 — 多人合作連線（multiplayer-coop）

> 性質：**規畫用設計文件**，非實作 spec。記錄整體架構、關鍵決策與子專案拆解，供未來動工時各子專案再各自展開 spec/bdd/acceptance/plan/tasks。
> 日期：2026-06-26　狀態：規畫中（未實作）

---

## 1. Overview

讓 **3–4 名玩家在同一張地圖一起合作生存**（朋友私房、非公開房間）。單人遊玩維持現有流程不變；
多人為獨立分支：主選單分「單人／多人」，多人可建立或加入房間，於等待室就緒後由房主開始。

核心技術路線：**確定性 lockstep**——網路上只傳「每位玩家每幀的輸入」，每台機器各自跑同一份
確定性模擬得到相同結果。引擎現有的固定步長（1/60）、種子化亂數（mulberry32）、純函式 system /
純資料 World 正好是此路線的地基。

## 2. Goals / Non-Goals

**Goals**
- 2–4 人同場、同一張地圖、合作生存。
- 單人流程零退化。
- 朋友私房：房間碼／邀請連結即玩，無公開房間/配對。
- 免費為前提的傳輸方案。
- 頻寬極小（只傳輸入）。

**Non-Goals（本期不做，列為未來）**
- 公開房間／配對／排行榜聯網。
- 房主遷移（host migration）——房主離線即結束本局。
- 復活（revive）——死亡後轉觀戰至全滅。
- 中途加入（開打後鎖房）。
- 對戰／PvP（僅合作）。
- 手機當控制器等跨裝置輸入。
- 防作弊（私房合作，不需權威防護）。

## 3. 為何現有架構適合

| 多人所需 | 現況 | 出處 |
|---|---|---|
| 跨機器狀態一致 | 固定步長、邏輯與 FPS 解耦 | `engine/Game.ts` |
| 可重現亂數 | 種子化 mulberry32，模擬不呼叫 `Math.random` | `engine/core/rng.ts` |
| 無隱藏狀態 | 純函式 system + 純資料 `World` | `engine/systems/*`, `engine/World.ts` |

→ 具備做 lockstep 的條件；最大工作不在「連線」，而在「把單人 World 改成多人 + 升級流程重設計 +
確定性稽核」。

## 4. 架構決策

### 4.1 同步模型：確定性 lockstep（只傳輸入）

- 每幀蒐集所有玩家輸入 → 廣播 → 各 client 以相同 tick 順序餵給各自的 `World.step()`。
- 子彈地獄場上實體上百，**逐幀同步實體狀態太吃頻寬**；只傳輸入則 payload 極小（4 人份按鍵）。
- 代價：需嚴格確定性（子專案 2）＋ 處理輸入延遲（輸入緩衝 / 固定延遲幀）。

### 4.2 傳輸層：託管免費服務當「輸入匯流排」

- 傾向 **Playroom Kit**（為瀏覽器遊戲設計、邀請連結、免費額度足夠 4 人、內建 NAT/STUN/TURN）。
- 替代：Photon 免費層、或 Cloudflare/PartyKit 免費層自架輕中繼、或 PeerJS P2P（需自理 TURN）。
- **關鍵**：無論選誰，只當「轉發輸入」用，不用其狀態同步去傳實體 → 廠商綁定低、日後可換。

### 4.3 主機角色：輕量房主協調

- 房主負責：產生房間碼、決定共用種子與地圖、蒐齊每幀輸入後定序廣播（tick 協調者）、按「開始」。
- 房主**不跑權威遊戲邏輯**（每台都自己算）；房主離線＝本局結束（無遷移）。

## 5. 關鍵設計決策（含建議值，未來可改）

- **D-1 升級流程（多人）**：**世界不暫停**。升級時僅該玩家畫面出現限時升級浮層；世界持續推進，
  其角色停在最後輸入（仍可被攻擊）；逾時（固定 N 幀）以**確定性規則自動選一張**，確保各機器一致。
  - 理由：lockstep 下所有人必須同步推進，不能為一人凍結全場。
  - 替代：「全場暫停、所有人一起選」——同步單純但 4 人頻繁升級會一直打斷彼此，體驗差。
  - **單人為特例**：維持現有「暫停選卡」（唯一玩家就是選的人，暫停無妨）。
- **D-2 角色選擇**：每位玩家在等待室各自選自己的角色。
- **D-3 地圖選擇**：由房主選一張，全房共用（本來就是「同一張地圖一起玩」）。
- **D-4 死亡處理**：玩家死亡 → 轉為觀戰（鏡頭跟隨存活隊友）；**全員死亡＝本局結束**。
- **D-5 難度調整**：生怪速率／敵人強度依玩家人數調整（沿用 `systems/spawn.ts` 曲線擴充）。
- **D-6 種子同步**：房主於開局把單一 seed 廣播給全房，確保各機器亂數序列一致。
- **D-7 玩家上限**：2–4 人；開打鎖房，滿房不可再加入。
- **D-8 鏡頭/HUD**：各 client 以自己的玩家為鏡頭中心；隊友顯示為同場 entity；HUD 顯示自己數值，
  隊友以簡化指示（小頭像/血條/箭頭）呈現。

## 6. UX 流程

```
主選單
├─ 單人遊玩 ─────────────────→ （現有流程：選角色/地圖 → 開打）
└─ 多人遊玩
   ├─ 建立房間 → 房間碼/邀請連結 → 等待室
   └─ 加入房間 → 輸入房間碼 ────→ 等待室
                                   ↓
   等待室：列出已加入玩家、各自選角色、就緒狀態；房主選地圖；房主按「開始」（鎖房）
                                   ↓
   遊戲中：同一張地圖、各自鏡頭、共享敵群；升級為非阻塞浮層（D-1）
                                   ↓
   結束：全員死亡 → 結算（可顯示各玩家戰績）→ 回主選單
```

## 7. 子專案拆解（未來各自走完整 SDD）

依「先做無網路也能驗證的、網路擺最後」：

### 子專案 1：多玩家 World + 升級流程重設計（純引擎、可 TDD、零網路）
- `World` 從單一 `player` 改為 `players: Entity[]`（各自輸入、HP、等級、loadout、升級佇列）。
- 共享敵群；生怪/碰撞/掉落對 N 玩家運作；難度依人數（D-5）。
- 升級流程改為 D-1 的非阻塞模型 + 逾時確定性自動選；單人退化為現有暫停特例。
- 死亡/觀戰狀態（D-4）。
- **交付**：可在單一程序內以 N 組輸入跑多玩家模擬，單元測試覆蓋；N=1 行為與現況一致。

### 子專案 2：確定性稽核與保證（純引擎、可測）
- 稽核模擬路徑：無 `Math.random`、`Date.now`、`performance.now`；物件/實體迭代順序固定；
  浮點來源一致（`Math.sin/cos/sqrt` 等使用點盤點，必要時集中包裝）。
- 建立**回放測試**：同一串（seed + 每幀輸入）→ 跑兩次/雜湊比對 → 結果完全相同。
- **交付**：lockstep 正確性護欄；CI 可跑的決定性回放測試。

### 子專案 3：本地多人 + 選單/等待室外殼（前端整合、無網路）
- 主選單加「單人／多人」分層；建立/加入/等待室 UI（先以本地假資料驅動）。
- 同機多輸入源（如鍵盤 + 另一組鍵位/手把）跑多玩家 World；多鏡頭或共享鏡頭、多/簡化 HUD。
- **交付**：完整多人體驗鏈路（選單→等待室→同場遊玩→結算）跑起來，但不接網路。

### 子專案 4：網路傳輸層（lockstep 輸入廣播）
- 接 Playroom（或選定的傳輸）：建立/加入房間、房間碼、等待室真連線、房主開始、種子廣播。
- 每幀輸入廣播 + 輸入緩衝/固定延遲幀；斷線/房主離開處理（D-1～D-7）。
- **交付**：子專案 3 的本地多人接上真網路，朋友可跨機同場。

**關係**：1 為地基；2 為 1/4 的正確性護欄；3 在 1 上把多人跑起來（含 UI 殼）；4 接網路。**先做 1。**

## 8. 風險與緩解

- **跨瀏覽器浮點不一致**（最大風險）：JS 雙精度 `+-*/` 多半一致，但超越函式（`sin/sqrt`…）可能有極小差異
  並隨步數累積發散 → 子專案 2 集中盤點、必要時自製確定性近似或定點化關鍵路徑；回放測試把關。
- **輸入延遲/抖動**：固定延遲幀（input delay）緩衝；私房小規模延遲可接受，暫不做 rollback。
- **斷線**：任一玩家斷線 → 暫停等候/標記離線；房主斷線 → 結束本局（D，無遷移）。
- **升級浮層期間角色受傷**（D-1）：以逾時自動選 + 短時保護或減速生怪緩解，數值待子專案 1 實機調。
- **免費額度變動**：傳輸層綁定低（只轉發輸入），可換；採用前確認當下額度。

## 9. 建議建置順序

1 → 2 →（1、2 穩固後）3 → 4。子專案 1、2 完全離線、可 TDD，風險最低，先把地基與正確性釘死；
3 把體驗跑起來；4 才引入網路與外部相依。每個子專案動工時各自走 brainstorming → spec/bdd/acceptance →
plan/tasks → 實作。

## 10. 開放問題（動工子專案 1 前再定）

- 升級浮層逾時的「確定性自動選」規則細節（選第一張？依權重？）與逾時秒數。
- 多人鏡頭：各自獨立鏡頭 vs 共享鏡頭（影響地圖移動範圍與走散處理）。
- 隊友走散：是否限制玩家間最大距離 / 畫面外指示。
- 難度依人數的具體曲線。
